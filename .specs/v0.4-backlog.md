## Note 1 — English hint = the `english` field (authoritative)

**Source:** user review 2026-06-28

**Current behavior:** The "English hint (optional)" input on `/add` is
forwarded to the AI as `note=`, and the AI's propose-labels response
overrides both `english` and `meaning` with its own drafts. The user's
typed text is treated as a hint, not a value.

**Desired behavior:** The English hint the user types should become the
`english` field directly — no AI override. The user is the sole author
of the canonical English gloss (this matches the SPEC §1.1 philosophy
"the user is the only author; the AI only proposes labels").

The `meaning` field can either:
  (a) also be authored by the user (they write a separate richer gloss),
  (b) still be AI-proposed, since the AI's contextual analysis adds value
      there.

**Implementation outline:**
- `/add/+page.svelte`: split the English input into a primary `english`
  field. Pre-fill `english` from the typed hint before showing the
  proposed form. The AI's `english` in the response is ignored or shown
  only as a "Compare with AI" alternative.
- Backend (`api/services/ai_client.py`): the system prompt can stop
  asking for `english` since the user owns it. Or keep asking and let
  the parser ignore it on the response side. Decide at spec time.
- Tests:
  - typing a hint and clicking Propose -> the form shows the user's hint
    in the english field, not the AI's draft
  - if the user clears the hint, the english field is empty and editable
  - the meaning field still shows the AI's proposal

## Note 3 — Antonyms are hanzi, not pinyin

**Source:** user review 2026-06-28

**Current behavior:** The AI returns `antonyms: ["bǎo"]` (pinyin-with-tones,
matching the word-unit id scheme from OQ2). The user finds pinyin harder
to scan than hanzi when reviewing the form — they want to see the actual
Chinese character.

**Desired behavior:** The `antonyms` field carries bare hanzi characters
(e.g. `["饱"]`). The frontend displays them as-is. Internally, on commit,
each hanzi is mapped to a word unit by:
  1. looking up `properties.hanzi == X` in existing word units, OR
  2. creating a new word unit with id = the pinyin of X (OQ2 still
     applies — word unit ids remain pinyin-with-tones).

The `connections` between the sentence and the antonym word unit use
the word-unit id (pinyin), not the hanzi. The hanzi in `antonyms[]`
is a label only.

**Schema change implication:** `CommitSentenceRequest.antonyms` should
accept list[str] of hanzi (which it already does — no schema change).
The route `commit_sentence.py` looks up each hanzi to find the word-unit
id, then writes the `opposite` connection using the id.

**Tests:**
- propose-labels returns `antonyms: ["饱"]` (hanzi) when expected
- commit-sentence with `antonyms: ["饱"]` correctly writes the opposite
  edge to the existing `bǎo` word unit (or creates a new one)
- the antonym field on the form displays hanzi
- the existing connector test `test_connector.py::test_opposite_edge_*`
  is updated to use hanzi in the request body

## Note 4 — Pinyin-on-hover + tone color coding

**Source:** user review 2026-06-28

**Feature:** When the user hovers over any hanzi character anywhere in
the UI, show its pinyin-with-tones as a tooltip. Each tone gets a
distinct color:
  - tone 1: red
  - tone 2: orange
  - tone 3: green
  - tone 4: blue
  - tone 5 (neutral): gray

**Why:** Chinese learners read hanzi but need to know pronunciation.
Hover eliminates the click-into-detail-page round trip. Tone colors make
the tone pattern visible at a glance — useful when scanning sentences
to memorize pronunciation.

**Implementation outline:**

1. **Backend:**
   - `GET /api/pinyin/{text}` — given a hanzi string, return a list of
     `{char, pinyin, tone}` objects. Backed by pypinyin with
     `style=Style.TONE3` (returns `ni3`, `hao3` etc; parse the trailing
     digit as tone 1-5).
   - Cached per-hanzi to avoid re-running pypinyin on every hover.

2. **Frontend:**
   - `app/src/lib/components/HanziWithPinyin.svelte` — wraps each char
     in `<span>` with:
       - `title={pinyin}` (native browser tooltip fallback)
       - a custom tooltip (better) showing pinyin + tone color bar
       - tone-colored underline or background
     - Fetches from `/api/pinyin/{text}` on mount, caches per-text.
   - Apply wherever hanzi is rendered:
     - `ResultRow.svelte` (search results)
     - `+page.svelte` (default page brand could be skipped)
     - `add/+page.svelte` (the hanzi textarea display after Propose)
     - `unit/[id]/+page.svelte` (hanzi in properties, sentence names in
       containing-sentences list)

3. **Tests:**
   - backend: `GET /api/pinyin/你好` returns `[{char:你, pinyin:nǐ, tone:3},
     {char:好, pinyin:hǎo, tone:3}]`
   - frontend: hover on a hanzi in ResultRow triggers the tooltip with
     correct pinyin and tone-colored class
   - tone colors are applied per character, not per word
