# v0.5 — Stable IDs, Bounded Tags, and Optional AI

**Status:** DRAFT
**Date:** 2026-06-29
**Scope:** Stable sentence IDs + dedup, de-AI toggle removal, dictionary-based word segmentation, bounded tag vocabulary, strict 2-call budget with audit log.
**Supersedes:** `.specs/language-brain.md` (v0.3) — this document is additive on top of v0.4; the base spec remains in force except where overridden here.

---

## 1. Overview

v0.5 introduces stable sentence IDs (server-side monotonic integer counter, never reused, never colliding), removes the AI-as-optional-gate with graceful auto-degrade, introduces a bounded tag vocabulary with a saved-slug list, and replaces jieba-only segmentation with a forward-maximum-match dictionary split backed by `vault/_meta/word_dictionary.json`. A strict 2-call budget with audit logging prevents runaway AI usage.

**The four themes of v0.5:**
1. **Stable IDs (T0):** Monotonic integer counter — smallest possible scheme, zero derivation, zero collision, sort order = insertion order.
2. **AI always-on (T1):** No toggle. AI is always attempted. Any failure (network, HTTP, parse, timeout) triggers auto-degrade to the dictionary-only path.
3. **Dictionary segmentation (T2):** Forward-maximum-match over `word_dictionary.json` replaces jieba. The dict grows organically from AI output.
4. **Bounded tags + 2-call budget (T3, T4):** AI receives `existing_groups` and prefers reuse. A hard 2-call ceiling raises `AIBudgetExceeded` on the 3rd attempt. Every call is audited to `vault/_meta/ai_audit.jsonl`.

**Non-goals (v0.5):**
- Grammar as a separate unit type (parked for v0.6+)
- User-merge or user-split UI for mis-segmented words
- Batch re-segmentation of the existing vault (T0 migration only touches sentence ids; it does NOT re-segment existing sentences)
- The existing `_resolve_segmentation` + AI `word_refs` override is removed; segmentation is now fully dict-driven at commit time

## 2. Glossary

| Term | Definition |
|------|------------|
| **AIBudgetExceeded** | Exception raised by `LLMClient` when a 3rd AI call is attempted within a single commit request. Results in HTTP 503. |
| **auto-degrade** | The commit path catches any AI failure (network timeout, HTTP error, JSON parse failure, missing response key) and falls through to the dictionary-only code path, committing the sentence without AI-suggested groups or meaning enrichment. |
| **dict_fallback** | A `WordToken` produced when a character has no entry in `word_dictionary.json`. Its `gloss` is `null` and `source` is `"dict_fallback"`. These units are written to disk but flagged as incomplete. |
| **forward-maximum-match (FMM)** | The segmentation algorithm: at each position, take the longest token (up to MAX_WORD_LEN=4) that appears in the dictionary. If no token matches, emit the single character as a `dict_fallback`. |
| **parked** | A `WordToken` for a PARKED_HANZI particle (`了 的 吗 呢 吧 啊 嘛 啦`). These have `gloss: null` and `parked: true`. The AI prompt is told they are parked so it does not suggest meanings for them. |
| **saved_group_tags** | The canonical, version-controlled list of all group slugs ever confirmed in the vault. Stored at `vault/_meta/saved_group_tags.json`. The AI prompt receives this as `existing_groups`; any user-confirmed group is appended (idempotent). |

## 4. Tasks

### T0: Stable Sentence IDs (Monotonic Integer Counter)

**Goal:** Replace the frontend-derived pinyin-slug sentence id with a server-side monotonic integer counter. Eliminate silent duplicate collisions from the live vault.

**Problem (live vault):**
- Frontend slug from `pinyin.toLowerCase().replace(/[^a-z0-9]+/g, '-')` on `wǒ xiǎng chī` → `w-xi-ng-ch` (aggressive tone-strip + collapsing vowels with consonants).
- Three copies of "I like to eat" exist under `s-1`, `s-2`, `wo-xihuan-chi`.
- No collision detection: second commit silently overwrites the first.

**Solution: monotonic integer counter**
Sentence ids are monotonically increasing integers, never reused, never colliding. The id is the primary key for sentence units; other units reference sentences by id. The id is for the machine, not the user — the user edits through the frontend.

Why integer counter over alternatives:
- **Pinyin-slug:** caused the bug we're fixing; requires collision policy and tone-strip library.
- **Date+hash hybrid:** sortable but ugly.
- **Integer counter:** smallest possible scheme, zero derivation, zero collision, sort order = insertion order, lockable with `fcntl.flock`, restart-safe.

**Counter file:**
```
vault/_meta/sentence_counter.json  →  {"next": 42}
```
Created on first commit if absent. `next` is the next unused integer.

**`next_sentence_id(vault_root) -> int`:**
1. Open `vault/_meta/sentence_counter.json` with `fcntl.flock` shared lock for read.
2. Return the current `next` value.
3. Write `next + 1` back using `os.replace` (atomic on POSIX) with exclusive lock.
4. Return the original `next` value.

The `fcntl.flock` ensures two parallel requests get distinct ids. `os.replace` ensures the write is atomic even if the process crashes mid-write.

**Files to add:**
- `api/services/sentence_id.py` — `next_sentence_id(vault_root: str) -> int`. Reads counter with `fcntl.flock` shared lock, returns current value, writes incremented value back with `os.replace` for atomicity.
- `scripts/cleanup_duplicate_sentences.py` — one-shot cleanup (see below).
- `tests/api/test_sentence_id.py` — counter monotonicity, atomicity, restart-safety, concurrency.
- `tests/scripts/test_cleanup_duplicate_sentences.py` — auto-keep richest, archive rest, idempotent, dry-run.

**Files to modify:**
- `api/routes/commit_sentence.py` — call `next_sentence_id()` at the top; ignore `body.id` entirely.
- `app/src/routes/add/+page.svelte` — remove lines 121–125 (slug derivation); do not send `id` in `commitSentence` payload.
- `api/schemas.py` — `CommitSentenceRequest.id` is dropped entirely (backend assigns).

### T2: Dictionary-Based Word Splitting

**Goal:** Replace the current jieba-plus-AI-override segmentation with a forward-maximum-match algorithm driven by `vault/_meta/word_dictionary.json`. The dict starts from existing `USER_DICT` entries and grows organically from AI output.

**Files to add:**
- `api/services/word_dictionary.py` — `WordDictionary` class: read/write `vault/_meta/word_dictionary.json`, `segment(hanzi, sentence_pinyin) -> list[WordToken]`, `grow_entry(hanzi, pinyin)`, `MAX_WORD_LEN = 4`
- `vault/_meta/word_dictionary.json` — initial seed (see §5 for schema)
- `tests/api/test_word_dictionary.py` — all test cases in §7 + integration tests

**Files to modify:**
- `api/routes/commit_sentence.py` — replace `_resolve_segmentation` + `_pair_word_refs_with_hanzi` with `word_dictionary.segment()`
- `api/services/ai_client.py` — after a successful `propose_labels`, call `word_dictionary.grow_entry` for each token where the source is `dict_fallback`
- `api/services/segmenter.py` — deprecated; `lcut` and `lcut_for_search` are no longer called from the commit path (they may be kept for search-query tokenization only)

**WordToken dataclass shape:**
```python
@dataclass
class WordToken:
    hanzi: str           # the token text (1–4 hanzi)
    pinyin: str          # tone-marked pinyin for the token
    source: str           # "dict" | "dict_fallback" | "parked"
    parked: bool = False  # True for 了 的 吗 呢 吧 啊 嘛 啦
    gloss: str | None    # None for dict_fallback and parked
```

**Pinyin disambiguation rule (LOCKED):**
```
pick_pinyin(candidates: list[str], sentence_pinyin: str) -> str
1. If exactly one candidate's tone-marked form appears in sentence_pinyin → return it.
2. Otherwise return the first candidate (highest-frequency dict entry).
```
Example: `sentence_pinyin = "wǒ shòu bù liǎo"`; candidates for `了` = `["liǎo", "le"]`; `liǎo` appears in sentence_pinyin → return `liǎo`.

**Algorithm (pseudo-code):**
```
function segment(hanzi: str, sentence_pinyin: str) -> list[WordToken]:
    tokens = []
    i = 0
    while i < len(hanzi):
        matched = False
        for length = min(MAX_WORD_LEN, len(hanzi) - i) down to 1:
            candidate = hanzi[i : i + length]
            if candidate in dict:
                readings = dict[candidate]   # list of PinyinReading
                pinyin = pick_pinyin([r.pinyin for r in readings], sentence_pinyin)
                token = WordToken(
                    hanzi=candidate,
                    pinyin=pinyin,
                    source="parked" if candidate in PARKED_HANZI else "dict",
                    parked=(candidate in PARKED_HANZI),
                    gloss=readings[0].gloss if not token.parked else None,
                )
                tokens.append(token)
                i += length
                matched = True
                break
        if not matched:
            # Single char, no dict entry → dict_fallback
            char = hanzi[i]
            pinyin = pypinyin_lazy(char)   # single-char pypinyin fallback
            tokens.append(WordToken(
                hanzi=char,
                pinyin=pinyin,
                source="dict_fallback",
                parked=(char in PARKED_HANZI),
                gloss=None,
            ))
            i += 1
    return tokens
```
Complexity: O(n × MAX_WORD_LEN) where n = number of characters; dict lookup is O(1) via hash.

**Acceptance criteria:**
1. `word_dictionary.json` is created at `vault/_meta/word_dictionary.json` on first run if absent.
2. `segment("我受不了这个")` returns 3 tokens: `["我", "受不了", "这个"]`.
3. `segment("吃了饭")` returns 3 tokens: `["吃", "了", "饭"]` where `了` has `parked=True`, `gloss=None`.
4. A character not in the dict (e.g. `僻`) returns `WordToken(source="dict_fallback", gloss=None)`.
5. For a dict entry with multiple readings, `pick_pinyin` selects the one matching `sentence_pinyin` when available, else the first.
6. After a successful AI call, each `dict_fallback` token's pinyin is written to `word_dictionary.json` as a new entry.
7. `MAX_WORD_LEN = 4` is enforced: `segment("世界上最")` does not match a 5-char token.
8. All 10 edge cases in §7 pass.

**Test plan (reference §7):**
- EC1–EC10: parametrized test with each edge case
- `test_segment_unknown_char`: random CJK not in dict → `dict_fallback`
- `test_segment_max_word_len`: 4-char max, 5-char compound → 2+1 or 1+1+1
- `test_pick_pinyin_sentence_match`: `了` in `shòu bù liǎo` → `liǎo` not `le`
- `test_pick_pinyin_no_match`: fallback to first candidate
- `test_grow_entry`: new entry appended to dict; re-read yields same pinyin
- `test_parked_particles`: all 7 PARKED_HANZI return `parked=True, gloss=None`
- `test_empty_sentence`: returns `[]`

**Dependencies:** T1 (for auto-degrade integration)

**Estimated complexity:** L (the primary implementation task)

### `WordToken` dataclass

```python

    hanzi: str              # e.g. "受", "不了", "这个"

    source: str             # "dict" | "dict_fallback" | "parked"

    gloss: str | None = None  # None for dict_fallback and parked
```

## 6. Algorithm Details for Dictionary Split

### Forward-Maximum-Match (FMM)

```
function segment(hanzi: str, sentence_pinyin: str) -> list[WordToken]:
    tokens := []
    i := 0
    while i < len(hanzi):
        matched := false
        for length := MIN(MAX_WORD_LEN, len(hanzi) - i) down to 1:
            candidate := hanzi[i : i + length]
            if candidate in dict:
                readings := dict[candidate]          # list of PinyinReading
                pinyin := pick_pinyin(readings, sentence_pinyin)
                gloss := readings[0].gloss
                if candidate in PARKED_HANZI:
                    gloss := None
                tokens.append(WordToken(
                    hanzi=candidate,
                    pinyin=pinyin,
                    source="parked" if candidate in PARKED_HANZI else "dict",
                    parked=(candidate in PARKED_HANZI),
                    gloss=gloss,
                ))
                i += length
                matched := true
                break
        if not matched:
            char := hanzi[i]
            pinyin := pypinyin_lazy(char)            # single-char fallback
            tokens.append(WordToken(
                hanzi=char,
                pinyin=pinyin,
                source="dict_fallback",
                parked=(char in PARKED_HANZI),
                gloss=None,
            ))
            i += 1
    return tokens
```

### `pick_pinyin(candidates: list[str], sentence_pinyin: str) -> str`

```
function pick_pinyin(candidates: list[str], sentence_pinyin: str) -> str:
    for candidate in candidates:
        # Normalize: remove tones from sentence_pinyin for matching
        normalized_sentence := strip_tones(sentence_pinyin)
        normalized_candidate := strip_tones(candidate)
        if normalized_candidate in normalized_sentence:
            return candidate
    return candidates[0]   # highest-frequency fallback
```

**Complexity:** O(n × MAX_WORD_LEN) where n = number of characters in `hanzi`. Dict lookup is O(1) (hash map). `MAX_WORD_LEN = 4` limits the inner loop to 4 iterations.

## 7. The 10 Dictionary Edge Cases

| # | Input | Expected Output | Notes |
|---|---|---|---|
| EC1 | `我受不了这个` | `["我", "受不了", "这个"]` | Multi-char compound; `了` in `受不了` is `liǎo` |
| EC2 | `吃了饭` | `["吃", "了", "饭"]` | `了` as particle → `le` (neutral tone); `了` in `吃饭` as part of compound → `liǎo` disambiguated by sentence pinyin |
| EC3 | `了解` | `["了解"]` | 2-char compound in dict; single token |
| EC4 | `了不起` | `["了不起"]` | 3-char compound in dict |
| EC5 | `世界上最` | `["世界", "上", "最"]` | 4-char max; `世界` is 2-char in dict, `上` and `最` are single |
| EC6 | `僻` (unknown char) | `[WordToken(source="dict_fallback", gloss=None)]` | No entry in dict; single char fallback |
| EC7 | `的` (parked particle) | `[WordToken(parked=True, gloss=None, source="parked")]` | Parked; no gloss suggested |
| EC8 | `了吗呢` | `["吗", "呢"]` with both parked; `了` as particle (EC2 handles compound vs particle) | Multiple parked particles in one string |
| EC9 | `为了` | `["为了"]` | 2-char compound; `了` here is NOT the particle |
| EC10 | `现在几点` | `["现在", "几", "点"]` | Mixed: 2-char compound + 2 single chars |

**Pinyin disambiguation for EC1, EC2, EC9:**
- `受不住` → `shòu bù zhù` (note: `liǎo` does not appear)
- `受不了` → `shòu bù liǎo` (the compound entry `liǎo` matches)
- `吃了` → sentence_pinyin contains `le` → pick `le`
- `为了` → sentence_pinyin contains `liǎo` or no `le`? If `为了` is in dict with `wèile`, sentence_pinyin `wèile` matches → pick `wèile`

## Appendix: PARKED_HANZI

```python
PARKED_HANZI: frozenset[str] = frozenset({"了", "的", "吗", "呢", "吧", "啊", "嘛", "啦"})
```

These 8 characters are handled as single-token `WordToken` with `parked=True` and `gloss=None` regardless of context. They do NOT appear in `word_dictionary.json` with a `gloss` (or if they do, the gloss is ignored when `parked=True`).
