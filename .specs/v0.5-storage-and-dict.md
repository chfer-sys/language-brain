# v0.5 — Storage + Dictionary Integration

**Status:** LOCKED
**Date:** 2026-07-08
**Supersedes:** `.specs/v0.5-bounded-tags-and-optional-ai.md` (historical — design exploration only)
**Target branch base:** `kickoff/v0.4.1-english-search` (a8790cc)

---

## 1. Overview

v0.5 replaces the JSON-file-only storage layer with a dual-store architecture: SQLite as the runtime store and JSON files as a human-readable, git-tracked mirror. It also introduces a bundled dictionary (SUBTLEX-CH, ~33,000 entries) as the authoritative source for word and compound identity — pinyin and English gloss are discovered from the dictionary, not invented by the user or guessed by AI.

The **dictionary as source of truth** is the key philosophical change. Word and compound units (their id, pinyin, and English gloss) come from dictionary lookup. The user's creative contribution is sentence-level meaning: their intuition, notes, and context. Word creation is pure dictionary lookup — no AI guessing at the word level.

v0.5 ships across five sub-versions: v0.5.0 (this SPEC, architecture lock), v0.5.1 (storage layer), v0.5.2 (id migration), v0.5.3 (dictionary integration), v0.5.4 (antonym auto-mirror), and v0.5.5 (search parity + performance). Each sub-version is independently testable and shippable.

The previous draft at `.specs/v0.5-bounded-tags-and-optional-ai.md` explored a small user-curated dictionary, forward-maximum-match segmentation, strict 2-call AI budget, and integer counter sentence ids. That design has been superseded by the locked decisions below after discussion with the user.

## 2. Glossary

| Term | Definition |
|------|------------|
| **id** | A variable-width, globally unique token assigned to each unit at creation. Format: `{type_letter}{integer-as-string}` — no padding, no fixed width. The integer grows as counts increase. See §4 for full scheme. The id is the canonical token used in all `word_refs`, `members`, `antonyms`, and `connections[].to` fields. |
| **type prefix** | A single letter preceding the integer: `W` for word (atomic, 1 hanzi), `C` for compound (2–4 hanzi, non-compositional), `S` for sentence, `G` for group. |
| **sort_key** | An integer column on the `unit` table providing canonical ordering. All `ORDER BY` clauses use `sort_key`, never lexicographic on `id`. The integer equals the sequential count assigned at id-creation time. |
| **variable-width id** | An id where the numeric portion has no zero-padding. `W1`, `W42`, `W1000`, `W10000` are all valid. The width adapts as counts grow — no future migration needed. |
| **padding** | Zero padding. There is none in the v0.5 id scheme. The variable-width approach eliminates the cap entirely. |
| **dictionary** | The SQLite dictionary tables (`dictionary_source`, `dictionary_entry`, `word`, `word_in_source`) containing Chinese word entries from SUBTLEX-CH (~33,000 entries). Provides pinyin and English gloss for every word and compound in the vault. |
| **SUBTLEX-CH** | A Chinese word frequency dictionary based on film/TV subtitles. CC-BY license, attribution required: "Cai & Brysbaert, 2010". Full list (~33,000 entries) used. |
| **mirror** | A read/write copy of the SQLite data in a human-readable format. JSON files at `vault/units/{words,sentences,groups}/*.json` are one mirror; `vault/index/vault.dump.sql` (sqlite3 text export) is another. All three stores must stay in sync after every write. |
| **auto-mirror** | The antonym write path: when unit A is saved with `antonyms: [B]`, the system also writes the reciprocal `antonyms: [A]` on unit B in the same SQLite transaction. Both files always have the symmetric pair after the write completes. |
| **atomic transaction** | A SQLite transaction that groups multiple writes (e.g. unit update + antonym mirror update) into a single ACID unit. Either all writes succeed or none do. |
| **round-trip** | The property that data written to SQLite can be read back and serialized to JSON, and that JSON can be re-imported to SQLite, producing identical data. All v0.5 storage code must preserve round-trip fidelity. |

## 4. Node Types and ID Scheme (LOCKED — Option Z)

**Format:** `{type_letter}{integer-as-string}` — no padding, no fixed width. The integer just grows as the count increases.

| Type | Letter | Examples | Growth pattern |
|------|--------|----------|---------------|
| Word (atomic, 1 hanzi) | `W` | `W1`, `W42`, `W999`, `W1000`, `W10000` | Variable width |
| Compound (2–4 hanzi, non-compositional) | `C` | `C1`, `C42`, ... | Same |
| Sentence | `S` | `S1`, `S42`, ..., `S100000` | Same (no cap; designed for >1000) |
| Group | `G` | `G1`, `G42`, ... | Same |

**Id is the canonical token** used in all `word_refs`, `members`, `antonyms`, and `connections[].to` fields. The filename (pinyin slug for human readability) is a separate concept and is NOT the id. The id lives in `unit.id` in SQLite AND in `"id": "W1"` inside JSON files.

**SQLite schema addition:**
```sql
CREATE TABLE unit (
    id          TEXT PRIMARY KEY,    -- "W1", "S42", "C1000" (variable width)
    type        TEXT NOT NULL,
    sort_key    INTEGER NOT NULL,    -- 1, 42, 1000, 100000 (canonical integer)
    ...
);
CREATE INDEX idx_unit_sort ON unit(type, sort_key);
```

**ORDER BY rules:**
- All unit ordering uses `ORDER BY sort_key`, never lexicographic on `id`.
- The `id` string is used for display, AI prompts, and all cross-references.
- `sort_key` is the authoritative integer for pagination and sequencing.
- No migration ever needed. The id format adapts as numbers grow.

**Tradeoff — AI prompt token cost:**
- `W1` = 2 chars, `W1000` = 5 chars, `W10000` = 6 chars.
- `S1` = 2 chars, `S1000` = 5 chars, `S100000` = 7 chars.
- Token cost grows linearly with count but is bounded (~7 chars max for any realistic personal vault). Acceptable.
- The letter prefix still disambiguates type at a glance in AI prompts.

**Rationale for variable-width over padded:**
- No cap, no future migration when `W` count exceeds 9999 or `S` count exceeds 99999.
- Simpler schema — one column instead of a separate `sort_key` for ordering concerns.
- The variable-width scheme is the simplest that has no upper bound.

## 5. The Dictionary

### 5.1 Source

**SUBTLEX-CH** (full list, ~33,000 entries) is the only dictionary source imported at v0.5 launch. CC-BY license (attribution required: "SUBTLEX-CH: Cai & Brysbaert, 2010"). Frequency data is from film/TV subtitles — reflects real-world usage frequency. The multi-source schema (4 dictionary tables) is designed for future extensibility, but v0.5 ships with SUBTLEX-CH only. Additional sources (HSK, CC-CEDICT, custom lists) can be added later via the same `scripts/build_dictionary.py --source` CLI.

Estimated size: ~2 MB as loaded SQLite data.

### 5.2 The 4 Dictionary Tables

```sql
-- Named bundles with metadata, license, attribution, priority, enabled flag
CREATE TABLE dictionary_source (
    id            TEXT PRIMARY KEY,     -- "subtlex-ch", "hsk-1-6" (future)
    name          TEXT NOT NULL,        -- "SUBTLEX-CH Top 3000"
    version       TEXT NOT NULL,
    license       TEXT NOT NULL,        -- "CC-BY 4.0", "Public Domain"
    attribution   TEXT NOT NULL,        -- "Cai & Brysbaert, 2010"
    priority      INTEGER NOT NULL DEFAULT 100,  -- lower = higher priority for English
    entry_count   INTEGER NOT NULL,
    imported_at   TEXT NOT NULL,
    enabled       INTEGER NOT NULL DEFAULT 1,
    UNIQUE(id, version)
);

-- One row per (source, hanzi, pinyin) triple. No in-source duplicates.
CREATE TABLE dictionary_entry (
    id              INTEGER PRIMARY KEY AUTOINCREMENT,
    source_id       TEXT NOT NULL REFERENCES dictionary_source(id),
    hanzi           TEXT NOT NULL,
    pinyin          TEXT NOT NULL,
    english         TEXT,
    frequency       REAL,
    part_of_speech  TEXT,
    extra           TEXT,              -- JSON blob for source-specific fields
    UNIQUE(source_id, hanzi, pinyin)
);
CREATE INDEX idx_dict_entry_hanzi  ON dictionary_entry(hanzi);
CREATE INDEX idx_dict_entry_pinyin ON dictionary_entry(pinyin);

-- Consolidated word entity: one row per unique (hanzi, pinyin) across all enabled sources.
-- english comes from the highest-priority enabled source.
CREATE TABLE word (
    id              TEXT PRIMARY KEY,    -- "W1", "W2", ... (variable-width, Option Z)
    hanzi           TEXT NOT NULL,
    pinyin          TEXT NOT NULL,
    english         TEXT,                -- chosen from highest-priority enabled source
    frequency       REAL,                -- from highest-priority source
    first_seen_at   TEXT NOT NULL,
    sort_key        INTEGER NOT NULL,    -- canonical integer for ORDER BY
    UNIQUE(hanzi, pinyin)
);

-- Many-to-many: tracks which sources contributed each word
CREATE TABLE word_in_source (
    word_id         TEXT NOT NULL REFERENCES word(id),
    source_id       TEXT NOT NULL REFERENCES dictionary_source(id),
    entry_count     INTEGER NOT NULL DEFAULT 1,
    PRIMARY KEY (word_id, source_id)
);
```

### 5.3 English Gloss Priority

When multiple sources contribute English for the same `(hanzi, pinyin)` pair, the entry from the highest-priority enabled source wins. Priority is the `dictionary_source.priority` column (lower number = higher priority).

At v0.5 launch (SUBTLEX-CH only):

| Priority | Source | Notes |
|----------|--------|-------|
| 0 | User-authored | `unit.english` always overrides (not stored in `word` table) |
| 50 | SUBTLEX-CH | Frequency from film/TV |
| 100 | Custom user lists | Future extensibility |

### 5.4 Source Lifecycle

- **Enable/disable:** `UPDATE dictionary_source SET enabled = 0 WHERE id = 'subtlex-full'`. Disabled sources are excluded from consolidation lookups and search, but their rows in `dictionary_entry` are preserved.
- **Re-enable:** `UPDATE ... SET enabled = 1`. Re-runs consolidation for that source.
- **Remove (cascade):** `DELETE FROM dictionary_source WHERE id = 'subtlex-full'` cascades to `dictionary_entry` (via FK) and `word_in_source` (via FK). Words that were ONLY in this source: their `english` falls back to the next-highest-priority enabled source's English. Words with no other source: `english` becomes NULL; the `word` row persists for history.
- **No data loss on remove:** `dictionary_entry` rows are deleted, but `word` rows persist. Re-importing the same source restores entries.

### 5.5 Consolidation Algorithm

`scripts/build_dictionary.py --source <id>`:

```
1. Read source file (SUBTLEX-CH CSV).
2. INSERT OR IGNORE INTO dictionary_source (id, name, version, license, attribution, priority, entry_count, imported_at, enabled).
3. For each entry in source:
   a. INSERT OR IGNORE INTO dictionary_entry (source_id, hanzi, pinyin, english, frequency, ...).
   b. Check if (hanzi, pinyin) exists in word table.
      - If not: INSERT INTO word with the highest-priority source's english and frequency.
      - If yes: skip (word already consolidated).
   c. INSERT OR IGNORE INTO word_in_source.
4. Report: N new entries, M consolidated, K total.
5. Idempotent: re-running the same import is a no-op.
```

### 5.6 First-Run Flow

1. User runs `python scripts/build_dictionary.py --source subtlex-ch`.
   - Seeds `dictionary_source` + `dictionary_entry` + `word` tables with SUBTLEX-CH entries.
   - Does NOT create any `unit` rows. A word becomes a unit only when it first appears in a committed sentence.

2. User commits first sentence "我想吃" via UI.
   - Segmenter looks up each token in `word` table: 我 → W1, 想 → W2, 吃 → W3.
   - Creates `unit` rows for any words that don't have one yet.
   - Creates sentence unit S1 with `word_refs: ["W1", "W2", "W3"]`.

A word becomes a **unit** the first time it appears in a sentence you commit.

### 5.7 Word Not in Dictionary (v0.5.3 detail — forward reference)

If a user commits a sentence containing a hanzi not in the `word` table, the system logs a warning and creates a minimal placeholder `word` entry with `english = NULL` and `source = "unknown"`. The sentence is not rejected. The user can manually edit the word's pinyin/english after the fact.

### 5.8 Pinyin Reading Disambiguation

One `word` row exists per unique `(hanzi, pinyin)` combination. A hanzi with multiple readings — e.g. 了 (`le` = aspect particle, `liǎo` = complement) — appears as **two separate `word` rows**, each with its own id:

```
word: (hanzi="了", pinyin="le",   english="(particle)",  id=W42, frequency=...)
word: (hanzi="了", pinyin="liǎo", english="(complement)", id=W43, frequency=...)
```

The `UNIQUE(hanzi, pinyin)` constraint enforces this. SUBTLEX-CH includes all readings for a given hanzi, so each distinct reading gets its own row at import time. The disambiguation happens at sentence-commit time: the segmenter (assisted by AI) picks which `word` row's reading matches the sentence's context, and that row's id is placed in `word_refs[]`.

**At sentence-commit time (segmentation):**
```
For each hanzi token in the sentence:
    If exactly one word row matches the hanzi → use that id
    If multiple word rows match the hanzi (multiple readings):
        Ask AI to pick the correct reading for this context,
        OR fall back to the row with highest frequency (SUBTLEX-CH data)
    If no word row matches → create placeholder with source="unknown"
```

The `pick_pinyin` function from earlier designs is no longer needed as a separate abstraction — the disambiguation is implicit in the one-row-per-(hanzi,pinyin) schema.

### 5.9 English Field Resolution

`word.english` vs `unit.english` — which is displayed?

- `word.english` is the **dictionary's canonical English** for the `(hanzi, pinyin)` pair. Set at consolidation time. It never changes unless a higher-priority source is added.
- `unit.english` is the **unit's own English** (the displayed value in the UI). On word-unit creation from a dict lookup, it is initialized from `word.english`. The user can override it via the UI.
- `word_refs[]` in sentence units stores the `word` id (not the english), so the reading-specific english travels with the unit through the sentence's `word_refs`.

**Resolution rule:**
```
UI display:     COALESCE(unit.english, word.english)
Search indexing: COALESCE(unit.english, word.english)
```

The user-authored `unit.english` always takes precedence. If `unit.english` is empty/null, the UI falls back to the dictionary's `word.english`. This allows the user to personalize a word's translation while preserving the dictionary default as a fallback.

## 6. Storage Layout

### 6.1 SQLite Schema

```sql
-- Polymorphic node table: all 4 unit types in one table
CREATE TABLE unit (
    id          TEXT PRIMARY KEY,    -- 'W1', 'C1', 'S1', 'G1' (variable width, Option Z)
    type        TEXT NOT NULL,       -- 'word' | 'compound' | 'sentence' | 'group'
    sort_key    INTEGER NOT NULL,   -- canonical integer for ORDER BY
    name        TEXT NOT NULL,       -- hanzi (W/C/S) or slug (G)
    pinyin      TEXT,                -- NULL for groups
    english     TEXT,                -- NULL for groups
    meaning     TEXT,                -- user-authored sentence meaning; NULL for W/C/G
    properties  TEXT,                -- JSON blob for type-specific extra fields
    created     TEXT NOT NULL,       -- ISO date
    updated     TEXT NOT NULL,       -- ISO date
    author_confirmed INTEGER NOT NULL DEFAULT 1,
    FOREIGN KEY (id) REFERENCES unit(id) ON DELETE CASCADE
);
CREATE INDEX idx_unit_sort ON unit(type, sort_key);

-- Edge table: all 6 edge kinds
CREATE TABLE edge (
    id          INTEGER PRIMARY KEY AUTOINCREMENT,
    source_id   TEXT NOT NULL,
    target_id   TEXT NOT NULL,
    kind        TEXT NOT NULL,  -- 'word_of'|'group_member'|'antonym'|'lexical'|'semantic'|'group'
    score       REAL,
    FOREIGN KEY (source_id) REFERENCES unit(id) ON DELETE CASCADE,
    FOREIGN KEY (target_id) REFERENCES unit(id) ON DELETE CASCADE,
    UNIQUE(source_id, target_id, kind)
);

-- FTS5 for lexical search
CREATE VIRTUAL TABLE unit_fts USING fts5(
    id UNINDEXED,
    name,
    pinyin,
    english,
    content='unit',
    content_rowid='rowid'
);

-- sqlite-vec for semantic search (384-dim embeddings)
CREATE VIRTUAL TABLE vec_sentences USING vec(
    id TEXT PRIMARY KEY,
    embedding float[384]
);

-- Key-value bookkeeping
CREATE TABLE _meta (
    key    TEXT PRIMARY KEY,
    value  TEXT
);
```

**Index notes:**
- `idx_edge_source` on `edge(source_id)` for outgoing edge traversal.
- `idx_edge_target` on `edge(target_id)` for incoming edge traversal.
- `idx_edge_kind` on `edge(kind)` for filtered edge queries.
- `idx_unit_type` on `unit(type)` for type-filtered unit queries.
- `idx_unit_sort` on `unit(type, sort_key)` for ordered retrieval.

### 6.2 JSON Mirror

For each SQLite `unit` row, one JSON file is written to:

```
vault/units/words/{pinyin-slug}.json     -- type='word' or 'compound'

vault/units/groups/{slug}.json            -- type='group'
```

**File shape (sentence example):**
```json
{
  "id": "S1",
  "type": "sentence",
  "name": "我喜欢吃",
  "properties": {
    "hanzi": "我喜欢吃",
    "pinyin": "wǒ xǐhuān chī",
    "english": "I like to eat",
    "meaning": "expressing enjoyment of eating",

    "word_refs": ["wǒ", "xǐhuān", "chī"],
    "groups": ["G1"],
    "antonyms": []
  },
  "connections": [
    { "to": "S2", "kind": "lexical", "score": 1.0 },
    { "to": "S3", "kind": "semantic", "score": 0.81 }
  ],
  "created": "2026-07-08",
  "updated": "2026-07-08",
  "author_confirmed": true
}
```

**File shape (word example):**
```json
{
  "id": "W1",
  "type": "word",
  "name": "吃",
  "properties": {
    "hanzi": "吃",
    "pinyin": "chī",
    "english": "to eat",
    "groups": [],
    "antonyms": ["W15"]
```

**SQLite row → JSON field mapping:**
- `unit.id` → `"id"`
- `unit.type` → `"type"`
- `unit.name` → `"name"` (hanzi for W/C/S, slug for G)
- `unit.pinyin` → `properties.pinyin`
- `unit.english` → `properties.english`
- `unit.meaning` → `properties.meaning`
- `unit.properties` (JSON blob) → `properties` (merged)
- Edges are read from `edge` table and written to `connections` array

## 7. Edge Types (LOCKED)

| Edge kind | From | To | Direction | Computation |
|-----------|------|----|-----------|-------------|
| `word_of` | sentence | word or compound | outgoing | Computed at sentence commit: from `words[]` field |
| `group_member` | group | any unit | outgoing | Computed at group commit: from `members[]` field |
| `antonym` | word | word | symmetric | Written on word save; auto-mirrored (see §7 note) |
| `lexical` | sentence | sentence | symmetric | Computed: shared hanzi tokens in `words[]`; Jaccard score |
| `semantic` | sentence | sentence | symmetric | Computed: cosine similarity on `meaning` embeddings via sqlite-vec |
| `group` | sentence | sentence | symmetric | Computed: both share at least one group membership |

**Note on antonym auto-mirror:** When unit A is saved with `antonyms: [B]`, a SQLite transaction writes:
1. `INSERT OR REPLACE INTO edge (source_id, target_id, kind, score)` for A→B
2. `INSERT OR REPLACE INTO edge (source_id, target_id, kind, score)` for B→A
3. Updates both unit rows' `properties` JSON to include the updated antonyms list

Both files always have the symmetric pair after the transaction commits. Rollback on failure is automatic via SQLite's ACID guarantees.

## 8. The 5 Sub-Versions

### v0.5.1 — Storage Layer (SQLite + JSON Mirror)

**Goal:** Replace JSON-file-only storage with SQLite as runtime store, JSON files as mirror, and `vault.dump.sql` as git mirror.

**Files to add:**
- `api/services/db.py` — SQLite connection manager (path: `vault/index/vault.db`), schema init, migration runner
- `api/services/unit_store.py` — read/write units to SQLite + write JSON mirror atomically

**`scripts/build_dictionary.py` CLI:**
```bash
# Import a dictionary source (idempotent — re-running same source is a no-op)
python scripts/build_dictionary.py --source subtlex-ch --path ./subtlex-ch.csv

# List enabled sources
python scripts/build_dictionary.py --list

# Disable a source (keeps entries, excludes from lookups)
python scripts/build_dictionary.py --disable subtlex-full

# Remove a source (cascades to dictionary_entry, word rows persist)
python scripts/build_dictionary.py --remove subtlex-full
```

**`scripts/build_dictionary.py --source <id>` algorithm:**
```
1. Parse source file (format detected by file extension or explicit --format flag).
2. INSERT OR IGNORE INTO dictionary_source with id, name, version, license, attribution, priority, entry_count, imported_at, enabled=1.
3. For each parsed entry:
   a. INSERT OR IGNORE INTO dictionary_entry (source_id, hanzi, pinyin, english, frequency, part_of_speech, extra).
   b. Check if (hanzi, pinyin) exists in word table.
      - If not: INSERT INTO word with next sort_key, english from highest-priority source.
      - If yes: skip.
   c. INSERT OR IGNORE INTO word_in_source (word_id, source_id, entry_count=1).
4. Report: N new entries, M consolidated, K total.
5. Idempotent: second run with same source is a no-op (UNIQUE constraint prevents duplicates).
```

**WordToken dataclass:**
```python
@dataclass
class WordToken:
    id: str           # W1 or C1 from word table (variable-width, Option Z)
    hanzi: str        # the token text
    pinyin: str       # from dict (disambiguated by sentence_pinyin)
    english: str      # from dict
    source: str       # "dict" or "unknown"
    parked: bool      # True for 了 的 吗 呢 吧 啊 嘛 啦
```

**Segmentation algorithm (forward-maximum-match):**
```
segment(hanzi, sentence_pinyin):
    tokens = []
    i = 0
    while i < len(hanzi):
        for length = min(4, len(hanzi) - i) down to 1:
            candidate = hanzi[i:i+length]
            entry = word_table_lookup(candidate)
            if entry exists:
                reading = pick_reading(entry, sentence_pinyin)
                tokens.append(WordToken(
                    id=entry.id,
                    hanzi=candidate,
                    pinyin=reading.pinyin,
                    english=reading.english,
                    source="dict",
                    parked=(candidate in PARKED_HANZI),
                ))
                i += length
                break
        else:
            # Unknown character: log warning, create placeholder
            char = hanzi[i]
            tokens.append(WordToken(
                id=None, hanzi=char, pinyin=None, english=None,
                source="unknown", parked=(char in PARKED_HANZI)
            ))
            i += 1
    return tokens
```

**Pinyin disambiguation (`pick_reading`):**
```
pick_reading(word_row, sentence_pinyin):
    # word_row may have multiple pinyin variants (stored in extra JSON or separate table)
    # For now: single pinyin per (hanzi, pinyin) row in word table
    # Sentence pinyin disambiguates among multiple entries for same hanzi
    if strip_tones(word_row.pinyin) in strip_tones(sentence_pinyin):
        return word_row
    return word_row  # single reading; no frequency sort needed in new schema
```

## 10. Risks and Open Questions

### Risk 1: sqlite-vec maturity

`sqlite-vec` is pre-v1. Breaking API changes are possible between minor versions.

**Mitigation:** Pin `sqlite-vec` to an exact version in `pyproject.toml`. Wrap all vec operations in a thin adapter service (`api/services/vec_store.py`) so the vec-specific code is isolated from the rest of the storage layer. If `sqlite-vec` introduces a breaking change, only the adapter needs updating.

**Open question:** Which `sqlite-vec` version should we pin to? Does the current `pyproject.toml` already list it?

---

### Risk 2: SUBTLEX-CH attribution (CC-BY) and download URL stability

The dictionary is CC-BY. The attribution requirement ("SUBTLEX-CH: Cai & Brysbaert, 2010") is stored in `dictionary_source.attribution` and surfaced via `GET /api/meta/license`.

**Open question:** What is the stable URL for SUBTLEX-CH download? The original paper is at `https://www.bpselting.org/subtlex-ch` but the download URL may have changed. **Fallback for v0.5.3:** If the URL is unstable, ship the SUBTLEX-CH CSV file in the repo under `vault/_meta/sources/` (git-lfs or plain git if small) and require manual pre-population rather than download-at-first-run. This eliminates the network dependency entirely.

---

### Risk 3: "Word not in dictionary" — what does the system do?

The bundled dictionary covers ~6000–7000 entries. User input may contain rare characters not in the dict.

**Current behavior (v0.5.3):** A warning is logged, a placeholder unit with `source="unknown"` is created, and the user can manually edit the pinyin/english. The sentence is not rejected.

**Open question:** Should there be a UI indicator ("⚠ unknown word") on sentence detail views when a word has `source="unknown"`? Or should the system prevent sentence commit until all words are resolved?

---

### Risk 4: Concurrent antonym writes — transaction safety

Two threads simultaneously saving antonym pairs: Thread 1 saves A→[B], Thread 2 saves B→[A]. Without careful locking, one write could overwrite the other's changes.

**Current design:** A single SQLite transaction covers both the A→B and B→A edge inserts and the unit property updates. SQLite's transaction isolation prevents dirty reads; the `UNIQUE(source_id, target_id, kind)` constraint prevents duplicate edges.

**Open question:** Do we need a distributed lock (e.g., per-unit advisory lock) for the antonym write, or is the SQLite transaction sufficient? For v0.5 (single-user local app), SQLite's built-in isolation should be sufficient. If a multi-user scenario arises (v0.6+), revisit.

---

### Risk 5: FTS5 tokenizer for CJK — does `unicode61` give us what we want?

SQLite's FTS5 `unicode61` tokenizer handles CJK character segmentation via Unicode property settings. By default, it treats each CJK character as a separate token (which is acceptable for character-level matching but may not give optimal word-level search).

**Current design:** Use FTS5 `unicode61` tokenizer with `tokenize='unicode61 remove_diacritics 1'`.

**Open question:** Should we implement a custom FTS5 tokenizer that uses the dictionary for word-boundary detection? For v0.5, simple character-level FTS is sufficient (the search still works; it just may over-match). A custom tokenizer is a v0.6+ optimization.

---

### Risk 6: Dictionary file format drift

v0.5 ships with SUBTLEX-CH only (CSV format). The multi-source schema is designed for extensibility, but future sources will require additional parsers. Each source format (CSV, TSV, JSON, XML) requires its own parser module.

**Mitigation (v0.5):** Only one parser needed: `parsers/subtlex_csv.py`. Future source formats get their own module under `scripts/parsers/` following a common interface: `parse(file_path) -> list[dict]`. The `build_dictionary.py` CLI dispatches based on `--format` flag or file extension.

**Forward:** When adding HSK (JSON) or CC-CEDICT (text), create `parsers/hsk_json.py` and `parsers/cc_cedict_txt.py` respectively. The `dictionary_entry.extra` JSON column stores source-specific fields that don't map to the common schema.
