# SPEC v0.10 — SQLite Read Paths

**Status:** ready for grill
**Version:** v0.10
**Date:** 2026-07-24
**Author:** plan subagent
**Supersedes:** nothing (performance fix)
**Vault storage model:** v0.5 locked (SQLite + JSON files + SQL dump mirror; see `.specs/v0.5-storage-and-dict.md`)
**Branch:** `kickoff/v0.10-sqlite-read-paths`

---

## 1. Goal

Route the four hottest read paths through the existing SQLite `unit` and `edge` tables instead of scanning JSON files on disk. This eliminates O(n) and O(n²) file I/O that will hurt as the vault grows from 100 to 1000+ sentences. No new dependencies. No schema migration. No embedding/vector changes. No frontend work.

---

## 2. Context

A performance audit found that v0.5.1 built the SQLite layer (`vault/index/vault.db` with `unit`, `edge`, and `unit_fts` tables) but the read path still hits JSON files. The `migrate_json_to_sqlite.py` script populates SQLite from JSON, but:

1. **`write_unit` does NOT sync to SQLite** — it only writes JSON files. SQLite becomes stale after any write.
2. **All hot read paths call `list_units_by_type`** which globs and reads JSON files.
3. **The `edge` table IS populated by the migration** from JSON `connections[]` arrays, but is never queried.

This SPEC completes the v0.5 vision: SQLite as the runtime store, JSON as the git-tracked mirror.

---

## 3. Scope

### 3.1 Read paths to fix (4 total)

1. **`GET /api/vault/list` browse endpoint** — `api/routes/vault.py:87`
   - Current: `list_units_by_type(settings.vault, unit_type)` reads all JSON files of that type, filters by id prefix, sorts in-memory, paginates.
   - Target: single SQLite query `SELECT id, name, pinyin FROM unit WHERE type = ? ORDER BY sort_key LIMIT ? OFFSET ?`.

2. **`lexical_search` and `semantic_search`** — `api/services/search.py:573, 1013`
   - Current: `list_units_by_type(vault_root, "sentence")` loads all sentence JSON files to build the ranker input and to resolve FAISS hits.
   - Target: SQLite query to fetch sentence units by id list (for semantic) or all sentences (for lexical ranker input). The ranker itself stays pure; only the I/O layer changes.

3. **`GET /api/units/{id}` unit detail** — `api/routes/units.py:55, 167`
   - Current: `_sentence_ids_containing_word` scans all sentence JSON files; compound constituent lookup scans all word JSON files.
   - Target: SQLite query on `edge` table for word_of edges (sentence → word reverse lookup); SQLite query on `unit` table for compound constituent characters.

4. **`compute_connections` (read side only)** — `api/services/connector.py:1299, 1325, 1337`
   - Current: `list_units_by_type` for sentences, groups, words to compute edges.
   - Target: SQLite queries to fetch units for the algorithm layer. The write side (upserting edges to JSON) stays unchanged.

### 3.2 Out of scope

- **No new dependencies.** SQLite is stdlib. No SQLAlchemy, no async driver.
- **No schema migration.** The existing `unit`, `edge`, `unit_fts` tables are sufficient.
- **No embedding/vector changes.** FAISS index stays as-is. Semantic search still uses FAISS for the vector search; only the unit-resolution step changes.
- **No frontend work.** The API response shapes stay identical.
- **No changes to `compute_connections` write side.** It still writes edges to JSON files. The `edge` table is populated by the migration script and stays read-only for now.
- **No dual-write mechanism in this sub-version.** See §6 Risks for the rationale.

## 4. Acceptance Criteria

Each criterion is testable by qa-reviewer. All must pass.

- [ ] **AC1.** `GET /api/vault/list?type=sentence&limit=50&offset=0` performs ≤1 SQLite query and reads 0 JSON files. The response body is byte-identical to what the JSON path returned before this change (round-trip property).
- [ ] **AC2.** `lexical_search(vault_root, query)` performs ≤1 SQLite query to fetch sentences and ≤1 to fetch words (instead of globbing all JSON files). The ranked output (unit_ids, scores, order) is identical to the JSON-path baseline for the same query.
- [ ] **AC3.** `semantic_search(vault_root, query)` performs 1 SQLite query to resolve the FAISS hit unit_ids (instead of loading all sentence JSON files). The returned SearchHits are identical to the JSON-path baseline.
- [ ] **AC4.** `GET /api/units/{word_id}` for a word unit performs 1 SQLite query on the `edge` table to fetch containing_sentences (instead of scanning all sentence JSON files). The response body is identical to the JSON-path baseline.
- [ ] **AC5.** `GET /api/units/{compound_id}` for a compound unit performs 1 SQLite query to fetch constituent_characters (instead of scanning all word JSON files). The response body is identical to the JSON-path baseline.
- [ ] **AC6.** `compute_connections(vault_root)` reads units via SQLite (≤3 queries: sentences, groups, words) instead of JSON globs. The computed edges (lexical_pairs, semantic_pairs, group_pairs, opposite_pairs counts) are identical to the JSON-path baseline.
- [ ] **AC7.** Round-trip property: for every read path, the data returned via SQLite is identical to the data returned via JSON. Test: run the same query through both paths (JSON path kept as a fallback or test-only function) and assert deep equality.
- [ ] **AC8.** Performance smoke test: with 500 sentences in the vault, `GET /api/vault/list?page=1` performs ≤1 SQLite query and completes in <50ms (vs. >200ms for the JSON path). Assert query count via SQLite trace or mock.
- [ ] **AC9.** All existing tests pass. No regression in test count or coverage. The test suite must not import new dependencies.

## 5. Implementation Notes (sketch-level)

### 5.1 Browse endpoint (`api/routes/vault.py`)

Replace `list_units_by_type(settings.vault, unit_type)` with a SQLite query:

```python
conn = get_connection(settings.vault)
rows = conn.execute(
    "SELECT id, name, pinyin FROM unit WHERE type = ? ORDER BY sort_key LIMIT ? OFFSET ?",
    (unit_type, limit, offset),
).fetchall()
```

The `unit` table already has `type`, `name`, `pinyin`, and `sort_key` columns. The `idx_unit_type` and `idx_unit_sort_key` indexes cover this query. The response shape stays identical.

**Ponytail:** reuse `get_connection` from `db.py`. No new helper needed.

### 5.2 Search (`api/services/search.py`)

**Lexical search (line 573):** Replace `list_units_by_type(vault_root, "sentence")` with a SQLite query that fetches all sentences as dicts. The ranker (`lexical_rank`) is pure and stays unchanged. The assembler (`_assemble_hits`) needs the unit dict shape, so the SQLite query must return a dict with `id`, `properties.hanzi`, `properties.pinyin`, etc.

**Semantic search (line 1013):** Replace the `list_units_by_type` call that builds `sentences_by_id` with a targeted query: `SELECT id, name, pinyin, properties FROM unit WHERE id IN (?, ?, ...)`. The FAISS index returns unit_ids; we resolve them via SQLite instead of loading all sentences.

**Ponytail:** the SQLite query returns `sqlite3.Row` objects; convert to dict shape compatible with the existing ranker/assembler. One helper function `_unit_rows_to_dicts(rows)` handles the conversion.

### 5.3 Unit detail (`api/routes/units.py`)

**Containing sentences for word/compound (line 55):** Replace the JSON scan with a SQLite query on the `edge` table:

```python
SELECT source_id FROM edge WHERE target_id = ? AND kind = 'word_of'
```

This returns sentence ids that reference the word. The `idx_edge_target` index covers this query.

**Compound constituents (line 167):** Replace the JSON scan with a SQLite query:

```python
SELECT id, name FROM unit WHERE type = 'word' AND length(name) = 1 AND name IN (?, ?, ...)
```

The `name` column holds the hanzi. The query fetches single-character words whose hanzi is in the compound's character set.

**Ponytail:** reuse `get_connection`. No new helper needed.

### 5.4 `compute_connections` read side (`api/services/connector.py`)

Replace the three `list_units_by_type` calls (lines 1299, 1325, 1337) with SQLite queries that fetch units as dicts. The algorithm layer (`_compute_sentence_lexical_edges`, etc.) is pure and stays unchanged. The I/O layer (reading units) changes; the write layer (upserting edges to JSON) stays unchanged.

**Ponytail:** the same `_unit_rows_to_dicts` helper from §5.2 works here.

## 6. The O(n²) Connection Computation

**Decision: NOT in scope for this sub-version.**

The `compute_connections` function has two parts:
1. **Read side:** loads units to compute edges (O(n) file reads).
2. **Write side:** upserts edges to JSON files (O(n²) pair scan for lexical/semantic).

This SPEC fixes the read side (§5.4). The write side (the O(n²) pair scan) is a separate optimization that would require:
- Incremental computation (only recompute edges for the changed unit).
- Storing computed edges in the `edge` table (currently only populated by migration).
- A dual-write mechanism to keep `edge` in sync with JSON.

That's a larger change and warrants a separate sub-version (v0.11). For now, `compute_connections` reads via SQLite but still writes to JSON and still does the O(n²) scan.

**Tradeoff:** the read-side fix is low-risk and ships immediately. The write-side optimization is higher-risk (requires dual-write, incremental computation) and can wait.

## 7. Test Plan

Every PR ships with tests. The test plan has three layers:

### 7.1 Round-trip equality tests

For each read path, write a test that:
1. Runs the query through the JSON path (keep the old function as `_json_path_fallback` in test-only code, or call the old function directly in the test).
2. Runs the query through the SQLite path (the new implementation).
3. Asserts deep equality of the results.

Example: `test_vault_list_sqlite_matches_json` in `tests/api/test_vault_list.py`.

### 7.2 Performance smoke tests

For the browse endpoint, write a test that:
1. Populates a vault with 500 sentences (use `write_unit` in a loop).
2. Runs `migrate_json_to_sqlite` to populate SQLite.
3. Calls `GET /api/vault/list?type=sentence&limit=50&offset=0`.
4. Asserts the response completes in <50ms (or asserts ≤1 SQLite query via mock/trace).

Example: `test_vault_list_performance_500_sentences` in `tests/api/test_vault_list_perf.py`.

### 7.3 Existing-test regression

Run the full test suite (`pytest tests/`). Assert:
- No test failures.
- No new dependencies imported.
- Test count does not drop (regression = fewer tests).

**Ponytail:** the round-trip tests are the most important. If the SQLite path returns different data than the JSON path, that's a bug. The performance tests are secondary; they catch regressions but don't prove correctness.

## 8. Rollback

**Approach: pure cut-over, no fallback.**

The JSON read path is replaced entirely. If the SQLite path breaks, the fix is to revert the commit (git revert). There is no runtime toggle or fallback to the JSON path.

**Rationale:**
- The SQLite data is populated by the migration script, which is idempotent and can be re-run.
- If the SQLite path returns wrong data, the round-trip tests (§7.1) will catch it before merge.
- A runtime toggle adds complexity (YAGNI). If the cut-over breaks in production, revert the commit.

**Rollback steps:**
1. `git revert <commit-hash>` on the integration branch.
2. Re-run `migrate_json_to_sqlite` if the SQLite data is suspect (idempotent, safe).
3. Deploy the reverted branch.

## 9. Risks

### 9.1 Stale SQLite data

**Risk:** `write_unit` does NOT sync to SQLite. If a unit is created/updated/deleted, the SQLite `unit` table becomes stale. Reads via SQLite will return old data.

**Mitigation:** this SPEC assumes the operator re-runs `migrate_json_to_sqlite` after bulk writes (e.g., after a connection-update batch). For interactive edits (single-unit updates via `PUT /api/sentences/{id}`), the SQLite data will be stale until the next migration run.

**Future fix (out of scope):** add dual-write to `write_unit` so SQLite stays in sync on every write. That's a separate sub-version (v0.11 or later).

**Why not in this SPEC:** dual-write is a larger change. It requires:
- Opening a SQLite connection on every `write_unit` call (performance cost).
- Handling partial failures (JSON write succeeds, SQLite write fails).
- Testing the dual-write path (more test surface).

For now, the operator can re-run the migration script after bulk writes. This is acceptable for a solo-dev vault.

### 9.2 Git-diffability of vault data

**Risk:** the JSON files are the git-tracked mirror. If SQLite becomes the source of truth, the JSON mirror may drift.

**Mitigation:** JSON stays the source of truth. SQLite is a read-optimized index. The migration script syncs JSON → SQLite. The write path (`write_unit`) still writes JSON. This SPEC only changes the read path.

**Round-trip property:** the data returned via SQLite must be identical to the data returned via JSON (§4 AC7). If it's not, that's a bug.

### 9.3 Token cost in AI prompts

**Risk:** if the SQLite path returns different data than the JSON path, AI prompts may receive stale or incorrect context.

**Mitigation:** the round-trip tests (§7.1) assert deep equality. If the SQLite path returns different data, the test fails before merge.

### 9.4 Compound `topProps()` gap

**Risk:** the existing `topProps()` function in `app/src/routes/unit/[id]/+page.svelte` doesn't handle `compound` type (see AGENTS.md "Lessons crystallized"). This SPEC doesn't fix that.

**Mitigation:** out of scope. The compound read path (§5.3) works correctly; the frontend rendering gap is a separate issue.

### 9.5 `edge` table staleness

**Risk:** the `edge` table is populated by the migration script from JSON `connections[]` arrays. If connections are recomputed (`compute_connections`), the JSON files are updated but the `edge` table is not. Reads via SQLite will return stale edges.

**Mitigation:** same as §9.1. The operator re-runs the migration script after `compute_connections`. Future fix: dual-write or incremental edge sync.
