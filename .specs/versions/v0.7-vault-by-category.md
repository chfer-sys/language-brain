# SPEC v0.7 — Browse Vault by Category

**Status:** ready for grill
**Version:** v0.7
**Date:** 2026-07-15
**Author:** plan subagent
**Supersedes:** nothing (new feature)
**Vault storage model:** v0.5 locked (SQLite + JSON files + SQL dump mirror; see `.specs/v0.5-storage-and-dict.md`)

---

## 1. Goal

After this ships, the user can open `/vault` in a browser and see their entire
vault organised under three (or four) tabs — one per category: **Word**,
**Compound**, **Sentence**, and optionally **Group**. Each tab shows a
flat, scrollable, optionally-sorted list of units in that category with enough
context (hanzi, pinyin, id) to orient the user without opening a unit. This is
a read-only browsing view; editing any unit is out of scope for v0.7.

---

## 2. User Journey

1. User clicks **"Browse vault"** (new link added to the nav bar on `/`).
2. Browser navigates to `/vault` — a new page.
3. The page renders three tab buttons: `[Word]  [Compound]  [Sentence]`.
   The **Sentence** tab is active by default (most content expected there).
5. The active tab's list loads immediately (no second click needed).
6. Each row shows: `S{n}  我流口水了  wǒ liú kǒu shuǐ le` — id, hanzi, pinyin.
7. Clicking a row navigates to `/unit/{id}` (the existing unit detail page).
8. (Removed in grill: per-row "Search for more like this" link. Cleaner UI;
   the unit detail page already has its own connection search if the user
   wants to find related units.)
9. Sorting: by id ascending (default) or by pinyin ascending (A→Z). The
   sort control is a single `<select>` above the list. (Removed in grill:
   sort by `created` — YAGNI at current vault size.)
10. Pagination: 50 items per page; prev/next controls appear **only when
    `total > 50`**. Below that threshold, the entire list renders at once
    with no controls visible.

### 3.3 New API endpoint (FastAPI)

```
GET /api/vault/list?type=sentence&limit=50&offset=0&sort=id
```

| Parameter | Type | Default | Description |
|---|---|---|---|
| `type` | `sentence\|word\|compound\|group` | **required** | Which category to list |
| `limit` | `int` | `50` | Items per page; range `[1, 200]` |
| `offset` | `int` | `0` | Zero-based skip; must be `>= 0` |
| `sort` | `id\|pinyin` | `id` | Sort field; `id` = alphanumeric ascending, `pinyin` = by `snippet` (pinyin) ascending A→Z |

**Response shape:**

```json
{
  "type": "sentence",
  "total": 142,
  "limit": 50,
  "offset": 0,
  "sort": "id",
  "items": [
    {
      "id": "S1",
      "name": "我流口水了",
      "snippet": "wǒ liú kǒu shuǐ le"
    }
  ]
}
```

**Invariant:** `name` = hanzi, `snippet` = pinyin. The response must never
contain `english` or `meaning` keys (payload hygiene, per AC20/AC21 from the
master SPEC).

**Error cases:**
- Missing or unknown `type` → HTTP 422.
- `limit` outside `[1, 200]` → HTTP 422.
- `offset < 0` → HTTP 422.
- Unknown `sort` value → HTTP 422.

## 4. Data Layer

The v0.5 storage model (`.specs/v0.5-storage-and-dict.md`) uses SQLite at
`vault/index/vault.db` as the runtime index plus JSON unit files. The actual
on-disk layout has three top-level subdirectories:

```
vault/units/
├── sentences/   ← S{n}.json (one file per sentence)
├── words/       ← W{n}.json (words) AND C{n}.json (compounds) in the SAME directory
└── groups/      ← slug.json or G{n}.json (one file per group)
```

The type of a unit in `words/` is distinguished by id prefix, not by a
separate field or subdirectory. The existing
`unit_writer.list_units_by_type()` already reads all units of a given type
from these JSON files, sorted by id, by filtering on the id prefix (`W` /
`C` / `S` / `G`).

**Pre-existing drift to fix in v0.7 (Phase A, step 1.5):**
- `api/schemas.py` defines `UnitType = Literal["sentence", "word", "group"]`
  (no `"compound"`).
- `api/services/unit_writer.py` `VALID_UNIT_TYPES = {"sentence", "word",
  "compound", "group"}` (4 types).
- v0.7 endpoint accepts `type=compound`, so `UnitType` literal MUST be
  extended to include `"compound"`. This is a one-line fix and unblocks
  the new endpoint, the Pydantic validator, and the existing
  `UnitTypeFilters` component on the frontend (which also lacks `compound`).
- Frontend `app/src/lib/components/UnitTypeFilters.svelte` should add a
  `compound` filter chip for consistency, though this is technically
  out-of-scope for the `/vault` route and may be deferred.

**No new storage file is required.** The `list_units_by_type` function can be
reused directly. Sorting by pinyin requires reading the `properties.pinyin`
field from each unit (available in the JSON); sorting by `created` requires
reading `created`. Both are O(n) passes — acceptable for vaults up to ~10,000
units (the current vault is expected to stay well below that).

**Idempotency:** the endpoint is purely read-only; no migration script needed.

## 5. Idempotency

Not applicable — no migration, no write path. The endpoint is stateless and
read-only. Re-running any part of the v0.7 pipeline any number of times
produces the same vault state.

## 6. Acceptance Criteria

- [ ] **AC1.** `GET /api/vault/list?type=sentence` returns HTTP 200 with a
  JSON body whose top-level keys are exactly `type`, `total`, `limit`,
  `offset`, `sort`, `items`.
- [ ] **AC2.** The `items` array contains objects with exactly `id`, `name`,
  `snippet` — no `english`, `meaning`, or other unexpected keys.
- [ ] **AC3.** `type=word` and `type=compound` each return HTTP 200 with
  their respective subsets (filtering by id prefix `W` vs `C`).
- [ ] **AC4.** `type=sentence&limit=10&offset=0` returns the first 10
  sentences by id ascending; `offset=1` returns items 11–20.
- [ ] **AC5.** `sort=pinyin` returns items sorted alphabetically by `snippet`
  (pinyin ascending A→Z). (`created` sort removed in grill; YAGNI at
  current vault size.)
- [ ] **AC6.** An unknown `type` value returns HTTP 422 with a descriptive
  error message.
- [ ] **AC7.** The SvelteKit `/vault` page renders three tab buttons
  (Word, Compound, Sentence) and the Sentence tab is active on load.
- [ ] **AC8.** Clicking the Word or Compound tab fetches the respective
  `GET /api/vault/list?type=word` or `type=compound` and renders the returned items.
- [ ] **AC9.** Each row in the list is clickable and navigates to
  `/unit/{id}` (the existing unit detail page).
- [ ] **AC10.** The sort `<select>` on `/vault` re-fetches the list with the
  new `sort` param when changed.
- [ ] **AC11.** Pagination controls (prev/next) appear when `total > 50`
  and correctly adjust the `offset` parameter. When `total <= 50`, no
  pagination controls are rendered.
- [ ] **AC12.** The hero section on `/` contains a "Browse vault" link to
  `/vault`.

---

## 7. Out of Scope

- **Editing any unit.** The detail page (`/unit/{id}`) remains read-only in
  v0.7; the edit flow is a separate future phase.
- **Deleting any unit.** Same deferred phase.
- **Creating new units from the vault view.** Same deferred phase.
- **Filtering the list by search query.** The browse view is category-first;
  full-text search is the `/` page.
- **Group tab.** Pending user decision (see Open Questions).
- **Any change to the storage model.** v0.5 SQLite + JSON layout is locked.

## 8. Open Questions

> All questions resolved by user (see below). This section is kept for
> traceability.

1. **Group tab.** Decision: **No** — Word / Compound / Sentence only. Group
   browsing is deferred to a future phase when it has its own dedicated UI.
2. **Route location.** Decision: **New `/vault` route.** Clean separation
   between browse and search concerns.
3. **Sort options.** Two modes (id, pinyin). `created` sort deferred — YAGNI
   at current vault size (~32 sentences, ~114 words/compounds, ~30 groups).
4. **Cross-link to search per row.** Decision: **Skip.** Cleaner UI; the
   unit detail page already has its own connection search if the user wants
   to find related units.
5. **Pagination.** Decision: **Auto-show only when `total > 50`.** Below
   that threshold, the list renders in one shot with no controls. API
   still supports `limit` and `offset` for forward-compatibility; the
   threshold is purely a UI affordance.
